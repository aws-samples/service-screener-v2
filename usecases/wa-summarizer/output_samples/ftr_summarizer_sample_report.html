IAM-002.1: Rotate IAM Access Keys Regularly

High Risk
+
Finding:

1 IAM user has access keys that have not been rotated in over 90 days. Regular credential rotation is a security best practice to limit the exposure window if credentials are compromised.

Risk if Not Remediated:

Long-lived access keys create extended windows of vulnerability. If an access key is compromised without your knowledge (through code repository exposure, log files, compromised developer workstation, or insider threat), the attacker can maintain access indefinitely until the key is rotated. The longer a key remains active, the higher the probability of compromise and the greater the potential damage. Compromised credentials are used in 80% of data breaches, with average detection time of 207 days. Regular rotation limits this exposure window to a maximum of 90 days, significantly reducing risk. Additionally, stale keys may indicate the user is no longer actively using AWS, suggesting the account itself should be reviewed for necessity.

Implementation Plan:
  1. Identify all applications and scripts using the old access key
  2. Create a new access key for the user (users can have maximum 2 keys)
  3. Update all applications/scripts with the new access key credentials
  4. Test all integrations thoroughly to ensure they work with new credentials
  5. Monitor CloudTrail logs for any usage of the old access key
  6. After 24-48 hours of successful operation, deactivate the old access key
  7. Monitor for any errors or failed authentication attempts
  8. After 7 days with no issues, permanently delete the old access key
  9. Document the rotation in your security log
  10. Set up AWS Config rule or CloudWatch alarm to alert on keys older than 90 days
AWS CLI Implementation:
# List current access keys for user
aws iam list-access-keys --user-name aws-example-awscli

# Create new access key
aws iam create-access-key --user-name aws-example-awscli

# Output will show AccessKeyId and SecretAccessKey - save these securely
# Update your applications with the new credentials

# After testing, deactivate old key (replace OLD_KEY_ID with actual key)
aws iam update-access-key \
    --user-name aws-example-awscli \
    --access-key-id OLD_KEY_ID \
    --status Inactive

# Monitor CloudTrail for any usage of old key
aws cloudtrail lookup-events \
    --lookup-attributes AttributeKey=Username,AttributeValue=aws-example-awscli \
    --max-results 50

# After verification period, delete old key
aws iam delete-access-key \
    --user-name aws-example-awscli \
    --access-key-id OLD_KEY_ID

# Verify only new key exists
aws iam list-access-keys --user-name aws-example-awscli

  • IAM Best Practices - Rotate Credentials

NETSEC-001.1: Restrict Sensitive Ports in Security Groups

High Risk
+
Finding:

5 security groups have sensitive ports (SSH, RDP, MySQL, PostgreSQL, MongoDB, MSSQL, Oracle, NFS, DNS, SMTP) open to all IP addresses (0.0.0.0/0). This creates a significant attack surface for brute force attacks and unauthorized access attempts.

Risk if Not Remediated:

Exposing sensitive ports to the entire internet is one of the most critical security vulnerabilities. Attackers continuously scan the internet for open ports and launch automated attacks within minutes of discovery. Specific risks include: (1) SSH/RDP brute force attacks attempting to guess credentials, (2) Database exploitation through known vulnerabilities or weak passwords, (3) Data exfiltration if attackers gain access to databases, (4) Ransomware deployment through compromised RDP sessions, (5) Lateral movement within your network after initial compromise. According to security research, SSH brute force attacks occur within 24 hours of port exposure, with thousands of attempts per day. The average cost of a successful breach through this vector is $3.86M. Additionally, this configuration violates PCI DSS, HIPAA, and most compliance frameworks.

Implementation Plan:
  1. Audit current security group rules and identify legitimate source IPs/ranges
  2. Document business justification for each required access
  3. For administrative access (SSH/RDP), implement bastion host or AWS Systems Manager Session Manager
  4. Update security group rules to restrict to specific CIDR blocks (office IPs, VPN ranges)
  5. For database access, ensure only application security groups can connect
  6. Test connectivity from legitimate sources before removing 0.0.0.0/0 rules
  7. Remove 0.0.0.0/0 rules during maintenance window
  8. Monitor CloudWatch Logs and VPC Flow Logs for any connection failures
  9. Implement AWS Config rule to prevent future 0.0.0.0/0 rules on sensitive ports
  10. Set up CloudWatch alarm for security group changes
AWS CLI Implementation:
# List security group rules to identify 0.0.0.0/0 access
aws ec2 describe-security-groups \
    --group-ids sg-01234abcdef56789a sg-1111abcdef9999a sg-2222abcdef0000a sg-3333abcdef1111a sg-5555abcdef2222a \
    --query 'SecurityGroups[*].[GroupId,GroupName,IpPermissions]' \
    --output json

# Example: Restrict SSH (port 22) to specific IP range
# First, add new rule with restricted access
aws ec2 authorize-security-group-ingress \
    --group-id sg-01234abcdef56789a \
    --protocol tcp \
    --port 22 \
    --cidr 203.0.113.0/24 \
    --description "SSH access from office network"

# Then, remove the 0.0.0.0/0 rule
aws ec2 revoke-security-group-ingress \
    --group-id sg-01234abcdef56789a \
    --protocol tcp \
    --port 22 \
    --cidr 0.0.0.0/0

# For database ports (example MySQL 3306), restrict to application security group
aws ec2 authorize-security-group-ingress \
    --group-id sg-1111abcdef9999a \
    --protocol tcp \
    --port 3306 \
    --source-group sg-app-security-group-id \
    --description "MySQL access from application tier"

aws ec2 revoke-security-group-ingress \
    --group-id sg-1111abcdef9999a \
    --protocol tcp \
    --port 3306 \
    --cidr 0.0.0.0/0

# Verify changes
aws ec2 describe-security-groups \
    --group-ids sg-01234abcdef56789a \
    --query 'SecurityGroups[*].IpPermissions'

# Set up AWS Config rule to prevent future violations
aws configservice put-config-rule \
    --config-rule file://restricted-ssh-config-rule.json

IAM-006.1: Limit Inline Policy Full Access to Single Service

High Risk
+
Finding:

6 IAM roles have inline policies granting full access (*:*) to one or more AWS services. This violates the principle of least privilege and creates unnecessary security risks.

Risk if Not Remediated:

Overly permissive IAM policies significantly increase the blast radius of compromised credentials or insider threats. Granting full access to a service when only specific actions are needed violates the principle of least privilege and creates multiple risks: (1) Compromised credentials can be used to perform any action within the service, including destructive operations, (2) Accidental misconfigurations or script errors can cause widespread damage, (3) Compliance violations as most frameworks require least privilege access, (4) Difficulty in auditing and understanding actual permissions, (5) Increased attack surface for privilege escalation. Additionally, inline policies are harder to manage and audit compared to managed policies. The average cost of an insider threat incident is $11.45M, and overly broad permissions are a contributing factor in 63% of such incidents.

Implementation Plan:
  1. For each affected role, analyze CloudTrail logs to identify actual API calls made (last 90 days)
  2. Use IAM Access Analyzer to identify unused permissions
  3. Create custom managed policies with only required permissions
  4. Test new policies in non-production environment
  5. Attach new managed policies to roles
  6. Monitor application logs for permission errors
  7. After 7-day validation period, remove inline policies
  8. Document the changes and rationale
  9. Implement AWS Config rule to prevent future inline policies with full access
  10. Establish policy review process (quarterly)
AWS CLI Implementation:
# List inline policies for a role
aws iam list-role-policies --role-name ap-southeast-2_example_Full-access

# Get inline policy document
aws iam get-role-policy \
    --role-name ap-southeast-2_example_Full-access \
    --policy-name PolicyName

# Use IAM Access Analyzer to identify unused permissions
aws accessanalyzer create-analyzer \
    --analyzer-name least-privilege-analyzer \
    --type ACCOUNT

# Generate policy based on CloudTrail activity (requires Access Analyzer)
aws accessanalyzer get-generated-policy \
    --service-last-accessed-details-arn arn:aws:iam::123456789012:role/ap-southeast-2_example_Full-access

# Create custom managed policy with least privilege
aws iam create-policy \
    --policy-name CustomLeastPrivilegePolicy \
    --policy-document file://least-privilege-policy.json \
    --description "Least privilege policy based on actual usage"

# Attach managed policy to role
aws iam attach-role-policy \
    --role-name ap-southeast-2_example_Full-access \
    --policy-arn arn:aws:iam::123456789012:policy/CustomLeastPrivilegePolicy

# After validation, delete inline policy
aws iam delete-role-policy \
    --role-name ap-southeast-2_example_Full-access \
    --policy-name PolicyName

# Verify final permissions
aws iam list-attached-role-policies --role-name ap-southeast-2_example_Full-access
aws iam list-role-policies --role-name ap-southeast-2_example_Full-access
